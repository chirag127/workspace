# Single .env.example per repo — same superset everywhere, no comments

## The rule

1. **Every repo** in the family — master + every site + every
   extension + every package — ships its own `.env.example` file at
   the repo root.
2. **Every `.env.example` lists the FULL family superset** of
   environment variables. Same set of `KEY=` lines in every repo.
   The canonical superset is at
   [`templates/.env.example`](../../templates/.env.example) in the
   master repo; new keys land there first, then propagate.
3. **No comments inside `.env.example`.** Per user instruction.
   Just `KEY=` lines, one per line, in the order defined by the
   master template.
4. **Tokens are shared across repos.** One Razorpay key, one Sentry
   DSN, one Doppler config, one Firebase project. The single source
   of truth is [Doppler](../services/secrets/doppler.md); local
   `.env` files are a runtime mirror per
   [`decisions/security/secrets-management-doppler.md`](../decisions/security/secrets-management-doppler.md).
5. **Per-repo deviation is forbidden.** A repo that doesn't consume
   a key still lists it (with empty value) — it just goes unused.
   This keeps `cp .env.example .env` working uniformly and avoids
   "which keys does THIS repo need?" archaeology.

## Why

- **One-line bootstrap.** Any repo can be cloned and started by a
  contributor (or a fresh-machine user, or a GH Actions runner)
  with `cp .env.example .env` followed by `doppler secrets download
  --no-file --format env > .env` (or filling values manually). No
  README hunting, no doc-site lookup.
- **No comments by user instruction.** The user said: *"there will
  be no comments in the dot DNV example file in any of the
  repositories"*. Per the
  [`self-update-rule.md`](./self-update-rule.md) +
  [`future-overrides-past.md`](./future-overrides-past.md), this is
  load-bearing.
- **Same superset everywhere = no per-repo schema drift.** A new key
  lands in the master template, then the same line gets appended to
  every `.env.example` in the family in a single fan-out commit. No
  "site A knows about FOO_KEY but site B doesn't" bugs.
- **Shared tokens fit the family's actual state.** One Razorpay
  account, one Sentry org, one Firebase project — sharing keys is
  the truth, not a workaround.

## What this means concretely

- **Master superset lives at [`templates/.env.example`](../../templates/.env.example).**
  When a new env var enters the family vocabulary, append it to the
  template first, then fan out the change to every repo's
  `.env.example` in one sweep.
- **`.env` is gitignored** in every repo (per
  [`rules/no-hardcoded-secrets.md`](./no-hardcoded-secrets.md)).
- **`.env.example` is committed** in every repo and contains only
  `KEY=` lines (empty values).
- **Tokens originate in Doppler.** `doppler secrets download
  --no-file --format env > .env` materialises a working `.env` from
  the source of truth.
- **GH Actions runners** read the same superset via Doppler → GH
  Secrets sync — no per-workflow env hardcoding.
- **CI checks** the existence of `.env.example` and that its key set
  matches `templates/.env.example` exactly (one-line diff, modulo
  ordering tolerance) on every PR. Drift fails the PR.

## What this rule does NOT mean

- **Not** "the `.env` file itself is shared via git." It is not. Per
  [`no-hardcoded-secrets.md`](./no-hardcoded-secrets.md), no actual
  secret value enters the repo.
- **Not** "every repo USES every key at runtime." It only LISTS them
  in the example; whether a key is actually read at runtime is the
  consumer's choice.
- **Not** an exemption from
  [`repos-work-independently.md`](./repos-work-independently.md).
  Each repo still must `pnpm install && pnpm build` after a `cp
  .env.example .env` (with empty values where unused).
